30def export(model, args, f, export_params=True, verbose=False, training=False, 

31 input_names=None, output_names=None, aten=False, export_raw_ir=False, 

32 operator_export_type=None, opset_version=None, _retain_param_name=True, 

33 do_constant_folding=True, example_outputs=None, strip_doc_string=True, 

34 dynamic_axes=None, keep_initializers_as_inputs=None, custom_opsets=None, 

35 enable_onnx_checker=True): 

36 r""" 

37 Export a model into ONNX format. This exporter runs your model 

38 once in order to get a trace of its execution to be exported; 

39 at the moment, it supports a limited set of dynamic models (e.g., RNNs.) 

40 

41 Arguments: 

42 model (torch.nn.Module): the model to be exported. 

43 args (tuple of arguments): the inputs to 

44 the model, e.g., such that ``model(*args)`` is a valid 

45 invocation of the model. Any non-Tensor arguments will 

46 be hard-coded into the exported model; any Tensor arguments 

47 will become inputs of the exported model, in the order they 

48 occur in args. If args is a Tensor, this is equivalent 

49 to having called it with a 1-ary tuple of that Tensor. 

50 (Note: passing keyword arguments to the model is not currently 

51 supported. Give us a shout if you need it.) 

52 f: a file-like object (has to implement fileno that returns a file descriptor) 

53 or a string containing a file name. A binary Protobuf will be written 

54 to this file. 

55 export_params (bool, default True): if specified, all parameters will 

56 be exported. Set this to False if you want to export an untrained model. 

57 In this case, the exported model will first take all of its parameters 

58 as arguments, the ordering as specified by ``model.state_dict().values()`` 

59 verbose (bool, default False): if specified, we will print out a debug 

60 description of the trace being exported. 

61 training (bool, default False): export the model in training mode. At 

62 the moment, ONNX is oriented towards exporting models for inference 

63 only, so you will generally not need to set this to True. 

64 input_names(list of strings, default empty list): names to assign to the 

65 input nodes of the graph, in order 

66 output_names(list of strings, default empty list): names to assign to the 

67 output nodes of the graph, in order 

68 aten (bool, default False): [DEPRECATED. use operator_export_type] export the 

69 model in aten mode. If using aten mode, all the ops original exported 

70 by the functions in symbolic_opset<version>.py are exported as ATen ops. 

71 export_raw_ir (bool, default False): [DEPRECATED. use operator_export_type] 

72 export the internal IR directly instead of converting it to ONNX ops. 

73 operator_export_type (enum, default OperatorExportTypes.ONNX): 

74 OperatorExportTypes.ONNX: all ops are exported as regular ONNX ops. 

75 OperatorExportTypes.ONNX_ATEN: all ops are exported as ATen ops. 

76 OperatorExportTypes.ONNX_ATEN_FALLBACK: if symbolic is missing, fall back on ATen op. 

77 OperatorExportTypes.RAW: export raw ir. 

78 opset_version (int, default is 9): by default we export the model to the 

79 opset version of the onnx submodule. Since ONNX's latest opset may 

80 evolve before next stable release, by default we export to one stable 

81 opset version. Right now, supported stable opset version is 9. 

82 The opset_version must be _onnx_master_opset or in _onnx_stable_opsets 

83 which are defined in torch/onnx/symbolic_helper.py 

84 do_constant_folding (bool, default False): If True, the constant-folding 

85 optimization is applied to the model during export. Constant-folding 

86 optimization will replace some of the ops that have all constant 

87 inputs, with pre-computed constant nodes. 

88 example_outputs (tuple of Tensors, default None): Model's example outputs being exported. 

89 example_outputs must be provided when exporting a ScriptModule or TorchScript Function. 

90 strip_doc_string (bool, default True): if True, strips the field 

91 "doc_string" from the exported model, which information about the stack 

92 trace. 

93 dynamic_axes (dict<string, dict<int, string>> or dict<string, list(int)>, default empty dict): 

94 a dictionary to specify dynamic axes of input/output, such that: 

95 - KEY: input and/or output names 

96 - VALUE: index of dynamic axes for given key and potentially the name to be used for 

97 exported dynamic axes. In general the value is defined according to one of the following 

98 ways or a combination of both: 

99 (1). A list of integers specifying the dynamic axes of provided input. In this scenario 

100 automated names will be generated and applied to dynamic axes of provided input/output 

101 during export. 

102 OR (2). An inner dictionary that specifies a mapping FROM the index of dynamic axis in 

103 corresponding input/output TO the name that is desired to be applied on such axis of 

104 such input/output during export. 

105 

106 Example. if we have the following shape for inputs and outputs: 

107 

108 .. code-block:: none 

109 

110 shape(input_1) = ('b', 3, 'w', 'h') 

111 and shape(input_2) = ('b', 4) 

112 and shape(output) = ('b', 'd', 5) 

113 

114 Then dynamic axes can be defined either as: 

115 (a). ONLY INDICES: 

116 dynamic_axes = {'input_1':[0, 2, 3], 'input_2':[0], 'output':[0, 1]} 

117 

118 where automatic names will be generated for exported dynamic axes 

119 

120 (b). INDICES WITH CORRESPONDING NAMES: 

121 dynamic_axes = {'input_1':{0:'batch', 1:'width', 2:'height'}, 

122 'input_2':{0:'batch'}, 

123 'output':{0:'batch', 1:'detections'} 

124 

125 where provided names will be applied to exported dynamic axes 

126 

127 (c). MIXED MODE OF (a) and (b) 

128 dynamic_axes = {'input_1':[0, 2, 3], 'input_2':{0:'batch'}, 'output':[0,1]} 

129 keep_initializers_as_inputs (bool, default None): If True, all the initializers 

130 (typically corresponding to parameters) in the exported graph will also be 

131 added as inputs to the graph. If False, then initializers are not added as 

132 inputs to the graph, and only the non-parameter inputs are added as inputs. 

133 This may allow for better optimizations (such as constant folding etc.) by 

134 backends/runtimes that execute these graphs. If unspecified (default None), 

135 then the behavior is chosen automatically as follows. If operator_export_type 

136 is OperatorExportTypes.ONNX, the behavior is equivalent to setting this 

137 argument to False. For other values of operator_export_type, the behavior is 

138 equivalent to setting this argument to True. Note that for ONNX opset version < 9, 

139 initializers MUST be part of graph inputs. Therefore, if opset_version argument is 

140 set to a 8 or lower, this argument will be ignored. 

141 custom_opsets (dict<string, int>, default empty dict): A dictionary to indicate 

142 custom opset domain and version at export. If model contains a custom opset, 

143 it is optional to specify the domain and opset version in the dictionary: 

144 - KEY: opset domain name 

145 - VALUE: opset version 

146 If the custom opset is not provided in this dictionary, opset version is set 

147 to 1 by default. 

148 enable_onnx_checker (bool, default True): If True the onnx model checker will be run 

149 as part of the export, to ensure the exported model is a valid ONNX model. 

150 """ 

151 

152 from torch.onnx import utils 

153 return utils.export(model, args, f, export_params, verbose, training, 

154 input_names, output_names, aten, export_raw_ir, 

155 operator_export_type, opset_version, _retain_param_name, 

156 do_constant_folding, example_outputs, 

157 strip_doc_string, dynamic_axes, keep_initializers_as_inputs, 

158 custom_opsets, enable_onnx_checker)
